# 정보 구조

정보 구조는 독자가 필요한 내용을 쉽게 찾고 이해하도록 문서의 범위, 순서, 제목, 연결 관계를 설계하는 일이다.

## 한 페이지에서는 하나만 다루기

한 페이지는 하나의 핵심 목표만 다룬다. 한 문서에 너무 많은 개념을 담으면 독자가 핵심을 파악하기 어렵고, 필요한 정보를 찾기 어려워진다.

체크리스트:

- 제목 깊이가 `####` 이상이면 문서를 나누는 것을 고려한다.
- 여러 개념을 개략적으로 소개해야 한다면 개요 페이지를 만들고 개별 문서로 연결한다.
- 한 페이지에서 두 개 이상의 핵심 주제를 다루면 문서를 분리한다.

좋지 않은 예:

```markdown
# React 사용법
## 컴포넌트 생성
### 컴포넌트 기본 구조
#### JSX 사용법
##### JSX 문법 세부사항
## 상태 관리
### 상태 정의
#### useState 사용법
```

좋은 예:

```markdown
# React 컴포넌트 생성하기
## 컴포넌트 기본 구조
## 상태 관리
## 라이프사이클 메서드
```

## 가치를 먼저 제공하기

기능이나 세부 설정보다 독자가 얻을 수 있는 가치를 먼저 설명한다. 독자는 "이 기능이 왜 필요한가?", "내 문제를 해결하는가?"를 먼저 알고 싶어 한다.

체크리스트:

- 도입부에서 독자가 얻을 이점이 명확한가?
- 배경 지식이나 부가 정보는 후순위에 배치했는가?
- 기능 나열보다 문제 해결 효과를 먼저 말했는가?

좋지 않은 예:

> 이 스니펫은 다양한 설정 옵션을 제공합니다. 먼저 `connection_timeout`, `retry_attempts`, `max_pool_size` 등의 매개변수를 수정하세요.

좋은 예:

> 이 스니펫을 사용하면 PostgreSQL 데이터베이스 연결 속도가 50% 빨라집니다.

## 효과적인 제목 쓰기

제목은 문서의 핵심 내용을 간결하고 명확하게 전달해야 한다.

체크리스트:

- 제목에 핵심 키워드를 포함한다.
- 같은 수준의 제목은 같은 문체와 패턴을 사용한다.
- 제목은 가능하면 30자 이내로 쓴다.
- 느낌표나 물음표보다 평서문을 사용한다.
- 단계가 있으면 숫자를 사용한다.

좋지 않은 예:

```markdown
# 에러를 해결하는 방법은?
```

좋은 예:

```markdown
# `NOT_FOUND_USER` 에러를 해결하는 방법
```

## 개요 빠트리지 않기

개요는 제목 바로 아래에 두고, 문서의 핵심 내용과 독자가 얻을 수 있는 결과를 요약한다.

체크리스트:

- 개요만 읽어도 문서 주제와 목적을 파악할 수 있는가?
- "이 문서를 읽으면 무엇을 할 수 있는가?"에 답하는가?
- 기술적 배경보다 핵심 정보를 먼저 제공하는가?

좋지 않은 예:

> 이 문서는 TypeScript의 유틸리티 타입을 소개합니다. Partial, Pick, Omit 등의 유틸리티 타입을 사용할 수 있어요.

좋은 예:

> TypeScript의 유틸리티 타입을 사용해서 객체 타입을 변형하는 방법을 알아봅니다. 유틸리티 타입을 활용하면 반복적인 타입 선언을 줄이고, 더 유연하게 타입을 관리할 수 있습니다.

## 예측 가능하게 하기

문서는 일관된 제목, 형식, 구조, 용어를 사용해야 한다. 예측 가능한 구조는 독자의 인지 부담을 줄이고 탐색을 쉽게 만든다.

체크리스트:

- 같은 수준의 제목과 소제목이 일관된 패턴을 따르는가?
- 기본 개념에서 세부 내용으로 점진적으로 확장되는가?
- 설명이 먼저 나오고 코드가 뒤따르는가?
- 용어가 일관되게 사용되는가?

좋지 않은 순서:

```markdown
# React에서 useEffect 사용하기
## 비동기 데이터 요청하기
## 기본적인 사용법
## 클린업 함수 활용하기
```

좋은 순서:

```markdown
# React에서 useEffect 사용하기
## 기본적인 사용법
## 비동기 데이터 요청하기
## 클린업 함수 활용하기
```

## 자세히 설명하기

독자는 문서를 처음 읽는다. 꼭 필요한 배경지식, 개념 정의, 동작 조건, 단위를 생략하지 않는다.

체크리스트:

- 새로운 개념이 등장하면 한두 문장으로 정의한다.
- 그 개념을 왜 알아야 하는지 또는 어디에 쓰는지 설명한다.
- 기능이 어떤 조건에서 어떻게 동작하는지 설명한다.
- 입력값, 상태, 시간에 따라 결과가 달라지면 그 차이를 말한다.
- 수치의 단위와 기준을 명시한다.

좋지 않은 예:

> `sessions[].duration`: 세션의 지속 시간을 나타냅니다.

좋은 예:

> `sessions[].duration`: 세션의 지속 시간으로, 사용자가 로그인을 유지한 시간을 의미합니다. 세션이 수동 로그아웃으로 종료된 경우에는 실제 이용 시간을 나타냅니다. 시간 초과로 자동 종료되었다면 마지막 활동 시점까지의 시간을 나타냅니다. 단위는 밀리초(ms)입니다.
