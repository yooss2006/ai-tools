# peer-clarify

모호한 생각, 아이디어, 요청을 짧은 동료식 피드백으로 선명한 요구사항으로 다듬는 Codex skill입니다. 바로 실행하기보다 대화로 목표, 방향, 기준, 피할 것을 좁힙니다.

## 언제 사용하나

- 아이디어는 있는데 방향이 아직 흐릿할 때
- 요구사항을 바로 구현하기 전에 기준을 세우고 싶을 때
- 친구, 동료, 선배처럼 짧게 피드백을 주고받고 싶을 때
- 좋은 점, 아쉬운 점, 보강할 점을 함께 보며 아이디어를 깎고 싶을 때

이미 실행 가능한 단순 작업에는 사용하지 않습니다. 그때는 바로 실행하는 편이 낫습니다.

## 동작 방식

`peer-clarify`는 한 번에 가장 큰 불확실성 하나만 다룹니다. 질문만 던지지 않고 짧은 판단, 이유, 좋은 점, 아쉬운 점, 대안을 함께 말합니다.

피드백은 무조건 반박하지 않습니다. 좋은 부분은 좋다고 말하고, 약한 부분은 보강 제안으로 다듬고, 별로인 부분은 이유를 붙여 선을 긋습니다. 반박은 필요할 때만 사용합니다.

## 대화 예시

```md
좋은데? 사용자가 바로 비교할 수 있게 하려는 방향은 살려도 될 것 같아.
다만 기준이 너무 넓어서 결과가 흐려질 수 있어.
먼저 가격, 시간, 유지보수 중 하나를 1순위로 잡는 게 좋겠어.
```

```md
이건 좀 걸려. 자동화를 먼저 만들면 실제 반복 작업이 뭔지 확인하기 전에 도구부터 커질 수 있어.
일단 수동으로 한 번 처리해보고, 반복되는 세 단계만 자동화하는 쪽이 낫겠어.
```

## 목표 산출물

대화가 끝나면 다음 네 가지가 남아야 합니다.

- 목표
- 최종 방향
- 중요하게 볼 기준
- 피하기로 한 것

종료 요청을 받으면 목록이 아니라 짧은 줄글로 결정된 최종 상태만 요약합니다.

## 사용 예

```text
peer-clarify로 이 아이디어 같이 다듬어줘.
```

```text
동료처럼 봐줘. 이 방향 괜찮아?
```

```text
요구사항으로 정리하기 전에 빈틈을 같이 봐줘.
```
