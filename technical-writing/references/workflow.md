# 작업 흐름

기술 문서 작성과 리뷰는 문서 유형, 정보 구조, 문장 품질 순서로 진행한다. 이 흐름은 한 페이지짜리 문서와 여러 페이지로 구성된 문서 세트에 모두 적용한다.

## 핵심 관점

테크니컬 라이팅은 기술적인 내용을 정확하고 효과적으로 전달하는 글쓰기다. 중요한 점은 독자가 문제를 해결하고 목표를 이루도록 돕는 것이다.

좋은 기술 문서를 만들려면 다음 세 가지를 확인한다.

1. 문서 유형이 독자의 목적에 맞는가?
2. 정보가 독자 입장에서 예측 가능하게 구조화되어 있는가?
3. 문장이 명확하고 구체적이며 일관적인가?

## 전체 문서 리뷰 절차

1. 독자와 목표를 확인한다.
   - 독자가 초심자인가, 경험자인가, 특정 문제를 해결하려는가?
   - 문서를 읽은 뒤 독자가 무엇을 할 수 있어야 하는가?

2. 문서 유형을 판단한다.
   - 유형이 불명확하면 `document-types.md`를 읽어라.
   - 하나의 문서에 여러 목적이 섞여 있으면 주 목적을 정하고, 필요하면 문서를 분리하거나 유형을 조합해라.

3. 정보 구조를 점검한다.
   - 목차, 제목, 개요, 정보 순서, 페이지 범위, 크로스링크를 본다.
   - 구조 문제가 있으면 `information-architecture.md`를 읽어라.

4. 문장을 다듬는다.
   - 주체, 간결성, 구체성, 자연스러운 한국어, 용어 일관성을 본다.
   - 문장 개선이 필요하면 `sentence-editing.md`를 읽어라.

5. 예시가 필요하면 보강한다.
   - before/after 비교나 유형별 샘플이 필요하면 `examples.md`를 읽어라.

## 여러 페이지 문서 세트 설계

여러 페이지로 구성된 문서는 먼저 독자에게 필요한 문서 유형을 나눈 뒤, 내비게이션 구조를 설계한다.

기본 구조 예시:

```text
docs/
├── get-started.md
├── tutorials/
├── how-tos/
├── explanations/
├── reference/
├── troubleshooting.md
└── glossary.md
```

설계할 때는 다음을 확인한다.

- 학습 중심 문서는 전체 흐름과 개요를 이해하도록 돕는다.
- 문제 해결 문서는 특정 목표나 오류 해결에 집중한다.
- 설명 문서는 개념과 원리를 깊게 이해하도록 돕는다.
- 참조 문서는 API, 옵션, 오류 코드처럼 빠르게 찾아볼 정보를 정리한다.
- 튜토리얼에서 자세한 설명이 필요하면 가이드나 참조 문서로 링크한다.
- 관련 문서 사이에 크로스링크를 넣어 독자가 필요한 시점에 깊은 정보로 이동할 수 있게 한다.

## 응답 방식

문서를 리뷰할 때는 문제만 나열하지 말고 수정 방향을 제시한다.

- 문서 유형이 맞지 않으면 더 적절한 유형과 이유를 말한다.
- 구조가 흐트러져 있으면 새 목차나 섹션 순서를 제안한다.
- 문장이 모호하면 수정 문장을 직접 제시한다.
- 원칙을 적용하지 않는 편이 더 나으면 예외 이유를 설명한다.
