# 문장 다듬기

기술 문장은 명확하고 이해하기 쉬워야 한다. 정보 구조가 좋아도 문장이 모호하면 전달력이 떨어진다.

## 문장의 주체를 분명하게 하기

기술 문서를 읽는 독자는 주로 개발자이고, 문서에 적힌 작업을 수행하는 사람도 개발자다. 문맥상 개발자가 행동의 주체임을 고려해 문장을 쓴다.

체크리스트:

- 도구나 기술 자체를 불필요하게 행동의 주체로 만들지 않는다.
- 독자가 할 수 있는 일을 중심으로 설명한다.
- 수동형 문장은 능동형으로 바꾼다.
- 단, 도구나 기술의 동작 자체를 설명할 때는 도구가 주체가 될 수 있다.

좋지 않은 예:

- 이 라이브러리는 데이터베이스 초기화를 수행해요.
- 애플리케이션이 실행되기 전에 설정이 완료되어야 합니다.

좋은 예:

- 이 명령어를 실행하면 데이터베이스를 초기화할 수 있어요.
- 애플리케이션을 실행하기 전에 설정을 완료하세요.

## 필요한 정보만 남기기

문장은 짧고 간결하게 쓴다. 한 문장에는 하나의 생각만 담는다.

체크리스트:

- 긴 문장을 나눌 수 있는가?
- 한 문장에 여러 아이디어가 섞여 있지 않은가?
- 핵심 메시지와 무관한 표현을 제거했는가?
- 문서 흐름을 설명하는 메타 담화를 줄였는가?

메타 담화 예:

- 앞서 설명했듯이
- 이제 알아보겠습니다
- 아시다시피
- 결론적으로
- 사실은

좋지 않은 예:

> 이 API를 호출할 때 요청 헤더를 포함해야 하며, 올바른 인증 정보를 제공해야 정상적으로 응답을 받을 수 있습니다.

좋은 예:

> 이 API를 호출할 때 요청 헤더와 인증 정보를 포함하세요.

## 구체적으로 쓰기

모호한 표현을 줄이고, 누가 무엇을 어디에 어떻게 하는지 구체적으로 쓴다.

체크리스트:

- 명사형 표현을 동사로 바꾼다.
- "가능성이 있다", "일부 경우", "필요할 수도 있다" 같은 모호한 표현을 줄인다.
- 누가, 무엇을, 어디에, 어떻게 하는지 쓴다.
- 실제 동작을 이해할 수 있게 설명한다.
- 수치와 기준을 제시한다.
- 참조되는 값이나 개념의 맥락을 설명한다.

좋지 않은 예:

> 코드 최적화 진행 후 배포 수행이 필요합니다.

좋은 예:

> 코드를 최적화한 후 배포하세요.

좋지 않은 예:

> 데이터가 많을 때는 성능이 저하될 수 있습니다.

좋은 예:

> 데이터가 10,000건을 넘으면 응답 시간이 1초 이상 걸립니다.

좋지 않은 예:

> 파일을 업로드하면 자동으로 저장됩니다.

좋은 예:

> 사용자가 파일을 업로드하면 서버에 자동으로 저장됩니다.

## 자연스러운 한국어 표현 쓰기

불필요한 한자어, 번역체 표현, 잡다한 표현을 줄인다. 한국어에서는 영어식 명사형 표현보다 동사 중심 표현이 자연스럽다.

체크리스트:

- 수행하다, 진행하다, 실시하다 같은 불필요한 한자어를 제거한다.
- "A를 통한 B" 같은 번역체를 자연스럽게 바꾼다.
- 동사에서 파생된 명사형 표현을 직접적인 동사로 바꾼다.

좋지 않은 예:

> API 키를 이용한 사용자 인증 처리가 완료된 후, 데이터베이스 접속 설정 진행이 가능합니다.

좋은 예:

> API 키로 사용자를 인증한 후, 데이터베이스에 접속하도록 설정할 수 있습니다.

## 일관되게 쓰기

기술 용어와 표현을 일관되게 사용한다. 같은 개념을 여러 방식으로 표현하면 독자가 혼란을 느끼고 검색도 어려워진다.

체크리스트:

- 공식 기술 용어와 공식 문서의 표기를 따른다.
- 같은 개념은 같은 표현으로 쓴다.
- 약어는 처음 등장할 때 풀어쓴 뒤 사용한다.
- 외래어 표기는 독자의 익숙함과 업계 사용 빈도를 고려한다.

좋지 않은 예:

> 파일을 추가하려면 '파일 선택' 버튼을 클릭하세요. 파일을 첨부한 후 '저장'을 누르면 업로드가 완료됩니다.

좋은 예:

> 파일을 업로드하려면 '파일 선택' 버튼을 클릭하세요. 파일을 업로드한 후 '저장'을 누르면 업로드가 완료됩니다.

약어 예:

- 좋지 않음: 이 기능은 SSR을 지원합니다.
- 좋음: 이 기능은 SSR(Server-Side Rendering)을 지원합니다.
